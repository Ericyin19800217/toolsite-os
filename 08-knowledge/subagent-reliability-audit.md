# 子 agent 可靠性排查报告

日期：2026-06-10

## 结论

本次排查没有发现单一的代码级错误。更可信的根因是组合问题：

```text
任务颗粒度过大
→ 子 agent 执行到文件写入或依赖安装后未能稳定收口
→ 主线程等待或恢复时只能看到残留摘要
→ 最终表现为“子 agent 卡住”
```

子 agent 不是不能用，但当前项目不适合把关键路径直接交给子 agent 自动推进。

## 已观察到的证据

### 1. 残留子 agent id 无法被运行时管理

当前上下文中残留了子 agent：

```text
019eaf7b-9342-73a0-bfa7-7f38849a9762 / Bacon
```

但 `wait_agent` 返回：

```text
not_found
```

这说明会话摘要里保留了历史 id，但当前 multi-agent 运行时已经不能管理它。压缩、恢复或线程生命周期切换后，子 agent 不能被当成稳定可恢复资源。

### 2. 项目复盘已有同类症状

MVP 实现复盘中已有记录：

```text
子代理执行时多次出现“文件已写入但不返回、不提交”的情况。
```

这表明问题不是单次偶发，而是在本项目执行过程中重复出现。

### 3. 实施计划对子 agent 的任务颗粒度偏大

`02-site-builds/plans/2026-06-10-dim-weight-calculator-implementation-plan.md` 第一行强制推荐 `subagent-driven-development`，但任务内容包含：

- 创建 Astro app 骨架。
- 写配置和 CSS。
- 写业务逻辑。
- 写测试。
- 安装依赖。
- 修构建错误。
- 多页面实现。
- 浏览器 QA。

这种任务更像完整项目推进，不是子 agent 最适合的“独立、明确、短闭环”任务。

### 4. Git 历史显示主线程最终接管并收口

MVP 最终形成了多个小提交，说明代码没有丢失，问题主要发生在：

```text
子 agent 写入部分成果
→ 没有稳定返回 DONE / 提交 / 汇报
→ 主线程检查 git status 后接管
```

## 根因分层

### 主要根因：任务边界不够小

子 agent 适合：

- 只读探索。
- 单文件或少量文件修改。
- 明确输入、明确输出、明确退出条件。
- 不依赖用户授权、不依赖长时间网络安装、不依赖复杂浏览器 QA。

本项目之前把“搭建一个完整工具站 MVP”的多个阶段交给子 agent，超过了它稳定收口的最佳范围。

### 次要根因：收口协议不够硬

之前子 agent 任务没有强制要求每次返回：

```text
STATUS: DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED
FILES_CHANGED:
TESTS_RUN:
COMMIT:
NEXT_RISK:
```

缺少这个协议时，子 agent 即使已经完成一部分，也可能继续尝试完善，导致用户看到“卡住”。

### 次要根因：生命周期不可恢复

当前证据显示，子 agent id 在上下文压缩或恢复后可能只剩摘要，不一定还能被 `wait_agent` 或 `send_input` 管理。

因此不能把子 agent 当成跨长会话的持久工作单元。

### 次要根因：权限和授权会放大卡住感

本项目中已经出现过 Git 写入 `.git/index` 需要授权、网络依赖安装、浏览器/插件可用性等问题。子 agent 如果遇到这类边界，可能无法像主线程一样及时和用户沟通授权。

## 以后怎么用

### 默认原则

```text
主线程负责关键路径和最终收口。
子 agent 只做短任务、旁路任务、低耦合任务。
```

### 适合派给子 agent 的任务

- 读一个文件夹并总结结构。
- 检查一个组件是否符合某个 spec。
- 为单个函数或模块补测试。
- 对一个明确页面做只读 UX review。
- 搜集某一类候选工具或竞品清单。

### 不适合派给子 agent 的任务

- 从 0 到 1 搭完整站点。
- 需要安装依赖和修构建链。
- 需要多次 Git 提交。
- 需要用户授权。
- 需要长时间浏览器 QA。
- 涉及同一批文件的并行修改。
- 任何主线程下一步立即依赖结果的任务。

## 强制使用协议

以后如果使用子 agent，任务 prompt 必须包含：

```text
你只负责这个小任务，不要扩展范围。
如果 10 分钟内不能完成，返回 BLOCKED，不要继续尝试。
不要等待完美实现，满足验收条件后立刻返回。

最终回答必须使用：
STATUS:
FILES_CHANGED:
TESTS_RUN:
COMMIT:
BLOCKERS:
NEXT_RISK:
```

主线程必须执行：

```text
派出前：确认任务不是关键路径阻塞项
等待时：继续做非重叠工作，不空等
返回后：先看 git status 和 diff，再相信汇报
异常时：不重复派同一任务，先接管检查文件
结束后：close_agent，避免占用并发和留下残留状态
```

## ToolSite OS 当前执行规则

在 ToolSite OS 项目里，子 agent 暂时降级为可选辅助能力：

```text
选题阶段：可用于旁路竞品/资料整理，但主判断由主线程完成。
设计阶段：可用于只读审查，不负责最终设计决策。
建站阶段：不负责从 0 到 1 主实现，只负责独立小模块。
QA 阶段：可用于并行检查，但主线程必须复验。
复盘阶段：可用于素材归纳，但主线程写入最终知识库。
```
