# Obsidian 知识库调用协议

## 目的

ToolSite OS 可以调用 Obsidian 知识库中的历史沉淀，但必须经过分析、引用和冲突检查。

不能把 Obsidian 内容当作默认正确结论，也不能只凭记忆复述。

## 已知知识库位置

根据启动快照，相关 Obsidian 知识库路径包括：

```text
/Users/yinbing/Documents/Eric Knowledge Base/
```

已参考过的材料类型包括：

- 视频笔记。
- OPC 业务评估报告。
- AI-built 应用上线与维护服务包。
- QA 服务验证方案。
- 第一性原理。
- OPC 公司定位。
- OPC 生存边界。
- OPC 不公平优势。
- AI 工具站矩阵流程。

## 何时调用

以下场景应该主动考虑调用 Obsidian：

- 判断一个方向是否符合 OPC 长期定位。
- 设计 B/C 端服务包、报价、交付流程。
- 复用已有的视频、报告、案例和 SOP。
- 比较当前判断与过去判断是否冲突。
- 需要第一性原理、商业闭环、服务化、自动化工作流等长期方法论。
- 用户明确提到“知识库”“之前沉淀”“Obsidian”“历史材料”。

## 调用步骤

```text
1. 明确本次问题需要哪类知识
2. 在 Obsidian 路径中搜索相关笔记
3. 读取最相关的少量文件
4. 提取可复用判断
5. 标明引用来源
6. 检查是否与当前项目结论冲突
7. 只把经过分析的结论沉淀进 ToolSite OS
```

## 引用格式

在项目文档中引用 Obsidian 知识时，使用以下格式：

```markdown
来源：`/Users/yinbing/Documents/Eric Knowledge Base/.../文件名.md`
提取判断：
- ...
用于本项目：
- ...
冲突检查：
- 无冲突 / 存在冲突，当前采用 ...
```

## 冲突处理

如果 Obsidian 历史判断与当前项目判断不同，不能直接覆盖当前结论。

处理方式：

```text
历史判断
→ 当前上下文
→ 差异原因
→ 当前采用结论
→ 后续验证条件
```

示例：

```text
历史判断：AI 工具站广告变现不适合作为第一现金流。
当前上下文：ToolSite OS 不以广告收入作为第一周成功标准，而是把工具站作为实验田。
差异原因：目标从“短期现金流”调整为“流程验证 + 内容素材 + 长期资产”。
当前采用结论：允许自建工具站矩阵，但第一阶段成功指标不是广告收入。
```

## 沉淀规则

- Obsidian 是长期个人知识网络，不直接替代项目内知识库。
- ToolSite OS 只沉淀与本项目相关、已经分析过的结论。
- 不把整篇 Obsidian 笔记复制进项目，只摘取结论、来源和适用范围。
- 如果结论会影响执行路径，必须写入 `08-knowledge/learnings.md` 或相关工作站文档。
