# 项目原则

## 全盘判断原则

关键环节不能只执行当前命令，必须主动思考：

```text
是否还有别的方案？
哪种方案更优？
当前选择对后续选题、建站、SEO、商业化、知识沉淀有什么影响？
```

这条原则适用于：

- 是否建立 Git 仓库。
- 是否建立项目知识库。
- 选题是否只看搜索量，还是同时看对标网站、广告价值、风险和可维护性。
- 数据采集是手动、半自动、官方账号还是第三方 API。
- 文档沉淀在项目内，还是沉淀进 Obsidian。

## 证据驱动原则

不让 AI 编造搜索量、流量、收入或广告收益。

所有判断需要标明：

- 数据来源。
- 检查日期。
- 目标地区。
- 查询词。
- 对标网站。
- 置信度。

## 风险分层原则

不一刀切排除金融、健康、税务等高价值领域，但必须区分工具型和建议型。

```text
Green：低风险工具，优先推进。
Yellow：高价值但需免责声明、公式来源和边界说明。
Red：高风险建议型内容，第一阶段暂不推进。
```

## 先自用再服务化原则

第一阶段先用自建工具站矩阵验证流程。

如果流程跑通，再转成：

```text
人工服务包
→ 半自动内部工作站
→ 客户自助工具
```

## 轻量基础设施原则

Git、项目知识库、台账和 Prompt 库是项目地基，不是后期装饰。

但第一版必须轻量：

- 能复用。
- 能维护。
- 能接入后续工作流。
- 不制造额外管理负担。

## 工具缺口主动发现原则

流程设计过程中，如果现有插件或 skill 不能覆盖某个关键环节，需要主动提出工具缺口。

处理方式：

```text
定义缺口
→ 判断是否高频
→ 提炼 GitHub 搜索关键词
→ 搜索候选工具
→ 比较维护状态、成本、集成难度和风险
→ 向用户提出补充建议
```

不应等用户指出工具不够用后才考虑补充。

## Obsidian 知识调用原则

项目需要调用 Obsidian 知识库时，必须分析和引用来源。

要求：

- 先搜索和读取相关笔记，再提取判断。
- 标明 Obsidian 文件路径。
- 说明该判断如何用于 ToolSite OS。
- 检查历史判断与当前项目结论是否冲突。
- 不把 Obsidian 内容未经分析地直接搬进项目。

## 关键节点复盘原则

ToolSite OS 从 2026-06-09 起正式开始。

每完成一个关键节点，必须复盘并更新 Obsidian 项目复盘日志：

```text
/Users/yinbing/Documents/Eric Knowledge Base/60 Reviews/Project Reviews/ToolSite OS 项目复盘日志.md
```

复盘必须记录：

- 走过的弯路。
- 踩过的坑。
- 不足和修正。
- 新形成的原则。
- 可作为自媒体选题的素材。

这份日志是未来自媒体起号的素材库，不只是项目记录。

## 设计闸门原则

页面类工具站不能只以“功能可用”和“构建通过”作为完成标准。

进入部署前必须经过设计闸门：

```text
产品语境
→ 页面信息架构
→ 视觉层级和可信感
→ 桌面端 / 移动端真实浏览器验收
→ 反 AI 味检查
→ 小批次修复
```

执行规则：

- 先 Research / 诊断，不直接大改。
- 每轮 UI 修复目标控制在 4-5 个以内。
- 写清不可破坏约束，例如不要破坏桌面端、移动端、核心计算逻辑和 SEO 页面结构。
- 浏览器里看真实页面，不只看代码和测试。
- 避免紫蓝渐变、emoji 图标、模板卡片堆砌、无来源数据和填充文案。

## 上线与 30 天观察原则

AdSense 免费工具站第一阶段的成功标准不是 7 天收入。

更合理的观察指标是：

```text
是否被 Google 收录
→ 是否有 impressions
→ 是否有人真实使用工具
→ 是否有反馈
→ 是否能沉淀成服务包、模板或客户获客小工具
```

上线前必须检查：

- About / Contact / Privacy / Terms 是否存在。
- sitemap.xml / robots.txt / canonical 是否完整。
- Search Console 是否提交。
- 页面是否有真实工具价值，而不是只有 AI 生成说明文。
- 是否避免自点广告、诱导点击、异常流量和垃圾外链。
