# Product

## Register

product

## Users

ToolSite OS serves two user layers.

The first layer is the end user of each tool site. For the first MVP, this means ecommerce sellers, marketplace sellers, cross-border sellers, warehouse operators, and shipping beginners who need to estimate dimensional weight before buying a shipping label or quoting a shipment.

The second layer is Eric as the operator of the ToolSite OS workflow. The project must produce reusable workstations, prompts, templates, QA rules, SEO routines, and review material that can later become service packages or internal tools.

## Product Purpose

ToolSite OS exists to turn AI-assisted tool-site production into a repeatable workflow:

```text
idea validation
→ MVP brief
→ build
→ design gate
→ QA
→ deployment
→ SEO submission
→ 30-day review
→ reusable service asset
```

The first visible product is a free multi-carrier dimensional weight calculator. Its job is to help users quickly understand dimensional weight, actual weight, billable weight, carrier divisor differences, and source confidence without pretending to quote final shipping costs.

Success means the first tool is usable, trustworthy, indexable, maintainable, and strong enough to become a reusable template for future calculator-style tool sites.

## Brand Personality

Trustworthy, practical, precise.

The interface should feel like a professional shipping or operations tool: calm, quick to scan, plain about uncertainty, and useful within seconds. It should avoid hype, decorative AI styling, and revenue fantasy. The product voice should be direct and evidence-aware.

## Anti-references

Avoid:

- Purple-blue AI gradients.
- Emoji icons as product UI.
- Generic card grids that look like AI templates.
- Fake metrics, unsupported revenue claims, or invented search-volume numbers.
- Overly rounded SaaS surfaces.
- Landing-page theatrics that slow down the calculator task.
- Legal, financial, tax, or carrier-rule certainty beyond verified public sources.

## Design Principles

1. Put the task first. The calculator and result should be visible before long explanation.
2. Make uncertainty explicit. Formula source, confidence, divisor, and estimate disclaimers are part of the product, not footnotes.
3. Use familiar product UI. Standard form controls, predictable labels, strong focus states, and readable result summaries beat novelty.
4. Design for repeat use. Sellers and operators may return with different package sizes, carriers, and units.
5. Leave a reusable trail. Every design, QA, SEO, and deployment decision should become a template for the next tool site.

## Accessibility & Inclusion

Target WCAG AA for text contrast, keyboard navigation, form labels, focus states, and mobile layout.

Reduced motion should be respected. Motion, if added, should clarify state changes only.

Inputs must show units clearly so new shippers do not confuse inches with centimeters or pounds with kilograms.
